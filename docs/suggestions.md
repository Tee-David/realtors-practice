1. The Scraper Needs to Become a First-Class Service
The scraper sits in /scraper but it's treated like a side tool. In the Nigerian market, the scraper is the product — it's where the data moat lives. Nigerian real estate sites like Property Pro, Nigeria Property Centre, and Jiji don't have public APIs. The scraper should have its own scheduling system (not just a script you run), a job queue backed by Redis/Upstash with retry logic and dead letter handling, and a deduplication pipeline. Properties get relisted across multiple sites — without entity resolution (matching the same 3-bedroom flat in Lekki across PropertyPro and NPC), you'll have duplicates polluting your search index.
Concretely: build a scraper orchestrator that runs on a schedule (cron or BullMQ), tracks which sites were last crawled and when, handles failures gracefully per-site, and writes to a staging table before promoting clean data to the main properties table. Also, Nigerian sites change their HTML constantly — consider storing raw HTML snapshots so you can re-parse without re-scraping when your selectors break.
2. The Geocoding Pipeline is Your Hardest Problem
The README mentions MapLibre/OpenStreetMap for maps. But Nigeria's addressing system is the real engineering challenge here. There are no standardized street addresses in most areas. Listings say things like "off Admiralty Way, Lekki Phase 1" or "behind Shoprite, Sangotedo." You need a normalized location taxonomy: State → LGA → Area/Neighborhood → Estate/Street, and a geocoding pipeline that can handle these informal descriptions.
My recommendation: build a location normalization layer that sits between your scraper and your database. Use the Google Maps Geocoding API as the primary geocoder (it handles Nigerian addresses better than Nominatim), fall back to What3Words or Plus Codes for areas with no proper addressing. Cache every geocoding result aggressively — you're paying per API call. Store both the raw scraped location string and the normalized coordinates plus taxonomy. Over time, this cache becomes its own dataset and competitive advantage.
Also, configure Meilisearch's geosearch properly: add _geo (lat/lng) to every property document, set it as a sortable and filterable attribute, and enable _geoRadius filtering so users can search "3-bedroom flats within 5km of Victoria Island." Meilisearch handles this natively and returns _geoDistance in results — surface that in the UI.
3. Meilisearch Needs Hybrid Search
Right now you probably have basic full-text search. But Nigerian property searches are often semantic — someone types "quiet estate with generator in Ajah" or "boys quarter in GRA." Meilisearch now supports hybrid search combining keyword matching with vector embeddings. Set up an embedding pipeline: when properties are indexed, generate embeddings from the concatenated title + description + features using an embedding model, store the vector alongside the document, and enable Meilisearch's hybrid mode. This handles the gap between what users type and how agents describe properties.
Also set up synonyms: "BQ" → "boys quarter", "sqm" → "square meters", "self-con" → "self-contained." Nigerian real estate has its own vocabulary that Meilisearch won't know out of the box.
4. The Backend Needs Rate Limiting, Caching, and API Versioning
From the README, the backend is Express with PostgreSQL. For a platform that'll eventually serve agents and buyers:
Add rate limiting (express-rate-limit + Redis store) — especially on your search endpoints and any scraper-facing APIs. Add response caching with Redis/Upstash: property listings don't change every second, so cache search results for 5-10 minutes. Version your API from day one (/api/v1/properties) — once agents or third parties integrate, you can't break their calls.
Also, the KOYEB_DEPLOYMENT.md file suggests you're deploying the backend separately from Vercel. Make sure you have health checks, graceful shutdown handling, and structured logging (pino, not console.log). If this is going to production at scale like Millenium-Potters did, you'll want to debug issues at 2am without SSH-ing into anything.
5. Data Integrity and the Trust Problem
Nigeria's real estate market has a massive fraud and data quality issue — this is well-documented. Your platform has a chance to differentiate by building trust signals into the data layer. Consider adding a verification_status field to properties (unverified, agent_verified, photo_verified, visited), a data_source field tracking where each listing came from, a last_seen_at timestamp updated by the scraper (stale listings are a huge pain point), and a price_history table for tracking changes over time.
This isn't just a nice-to-have — it's what separates a data aggregator from a platform agents will actually trust and pay for.
6. The Frontend Dashboard is Solid — But Needs Search UX
The live app shows a clean dashboard with revenue analytics, property categories, status analysis. That's well-executed. But the search page should be the star of the product. Implement search-as-you-type with Meilisearch's instant search (they have a React SDK), faceted filtering by state/LGA/property type/price range/bedrooms, and map-based browsing where clicking a cluster shows properties in that area. The Data Explorer and Scraper management pages visible in the sidebar are smart power-user features — that's the kind of thing that shows you're thinking about the platform as a tool for agents, not just a listing site.
7. Clean Up the Root Directory
This is minor but matters for credibility: there are loose files in the root — image.png, image-1.png, image-2.png, image-3.png, koyeb.tar.gz, test_engine.py. These should either be in their proper directories or gitignored. The CLAUDE.md, instructions.md, suggestions.md, updates.md are development artifacts — consider moving them to docs/ or a .claude/ directory. A clean root signals discipline.
8. Testing
There's a test_engine.py in the root but no visible test directory structure. For the scraper especially, you need tests: unit tests for HTML parsers (feed them saved HTML fixtures), integration tests for the geocoding pipeline, and end-to-end tests for the search → Meilisearch → results flow. The backend API should have at minimum route-level tests with supertest. You don't need 100% coverage — just test the things that would cost you data integrity if they broke.
